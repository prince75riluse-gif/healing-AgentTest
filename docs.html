📚 Healing Agent Test Page Documentation

Overview

This test page is designed to help QA engineers and automation developers test healing agent functionality. It provides a comprehensive set of UI elements with various locator strategies.

What is a Healing Agent?

A Healing Agent is an AI-powered system that automatically detects and fixes broken UI locators in test automation. When a test fails due to a changed or broken locator, the healing agent attempts to find the element using alternative strategies.

Test Sections

1. Static Elements

Tests basic locator fallback strategies. These elements have multiple locator options (ID, Class, Text) that can be used for healing.

  • Login Button: ID = loginButton
  • Sign Up Button: ID = signupButton
  • Submit Button: ID = submitButton
  • Cancel Button: ID = cancelButton

2. Form Inputs

Tests DOM scoring and similarity matching. These inputs can be found using placeholder text, labels, and structural relationships.

3. Dynamic Content

Tests AI recovery for dynamically generated elements. Click "Generate Dynamic Content" to create new elements with unique IDs.

4. Modal & Overlays

Tests healing in overlay contexts. The modal can be opened and closed, testing z-index and visibility scenarios.

5. Conditional Elements

Tests visibility changes and show/hide scenarios. The toggle button reveals or hides a conditional element.

6. Navigation Links

Tests link locators using href attributes and text content.

How to Use This Page

For Manual Testing:

  1. Open the page in your browser
  2. Interact with each element to verify functionality
  3. Use browser DevTools to inspect element properties
  4. Test dynamic content generation

For Automation Testing:

  1. Create test scripts targeting elements on this page
  2. Intentionally break locators (e.g., change ID values)
  3. Run tests with healing agent enabled
  4. Verify the healing agent finds elements using alternative strategies

Element Locator Strategies

Each element supports multiple locator strategies:

  • ID: Unique identifier (most reliable)
  • Class: CSS class names (can match multiple elements)
  • Text: Visible text content (user-facing)
  • XPath: DOM path (can be fragile)
  • CSS Selector: Style-based selection

Testing Best Practices

  • Test one healing level at a time
  • Start with simple scenarios (Level 1 - Static Heuristics)
  • Progress to complex scenarios (Level 3 - AI Recovery)
  • Document healing success rates
  • Monitor performance metrics

Common Test Scenarios

Scenario 1: ID Change

Original: id="loginButton"
Modified: id="brokenLoginButton"
Expected: Healing finds element using text="Login"

Scenario 2: Class Change

Original: class="auth-btn primary"
Modified: class="auth-button primary-btn"
Expected: Healing finds element using ID or text

Scenario 3: Multiple Changes

Original: All locators working
Modified: All locators changed
Expected: Healing uses DOM scoring/AI recovery
